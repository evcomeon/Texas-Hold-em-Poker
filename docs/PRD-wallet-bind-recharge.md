# Web3 钱包绑定与充值系统 PRD

## 一、功能概述

### 1.1 背景
用户需要通过加密货币（USDT/USDC）充值筹码。为确保资金安全和账户关联，用户必须绑定 Web3 钱包地址。

### 1.2 目标用户
- 已通过邮箱/Google/用户名登录但未绑定钱包的用户
- 已通过 Web3 钱包登录的用户（自动绑定）

### 1.3 核心价值
- 确保充值资金可追溯
- 防止充值地址被冒用
- 提供安全的资产管理

## 二、业务规则

### 2.1 钱包绑定规则

| 规则 | 描述 |
|------|------|
| **唯一性约束** | 一个钱包地址只能绑定一个用户账户 |
| **单地址绑定** | 一个用户账户只能绑定一个钱包地址 |
| **更换绑定** | 用户可以更换绑定的钱包地址，需重新签名验证 |
| **绑定状态** | 绑定后不可解绑，只能更换 |
| **Web3登录用户** | 通过钱包登录的用户自动绑定，无需额外操作 |

### 2.2 充值规则

| 规则 | 描述 |
|------|------|
| **前置条件** | 必须绑定钱包地址才能充值 |
| **支持代币** | USDT、USDC（ERC-20） |
| **最低金额** | 1 USDT/USDC |
| **汇率** | 1 USDT/USDC = 10,000 筹码（可配置） |
| **确认数** | 需要 12 个区块确认 |
| **到账时间** | 约 3-5 分钟（取决于网络拥堵） |

## 四、安全设计

### 4.1 签名验证机制

```
┌─────────────────────────────────────────────────────────────────────┐
│                       签名验证流程                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  前端                              后端                             │
│  ┌──────────┐                     ┌──────────┐                     │
│  │ 连接钱包  │                     │          │                     │
│  └────┬─────┘                     │          │                     │
│       │ POST /wallet/bind/nonce   │          │                     │
│       │──────────────────────────►│ 生成Nonce│                     │
│       │                           │ 存储Nonce│                     │
│       │ { nonce: "xxx" }          │ 设置过期 │                     │
│       │◄──────────────────────────│ (5分钟)  │                     │
│       │                           └──────────┘                     │
│       │                                                            │
│       │ personal_sign(nonce)                                       │
│       │ (用户在钱包中签名)                                          │
│       │                                                            │
│       │ POST /wallet/bind/verify                                   │
│       │ { address, signature }                                     │
│       │──────────────────────────►┌──────────┐                     │
│       │                           │ 验证签名 │                     │
│       │                           │ 检查Nonce│                     │
│       │                           │ 检查唯一 │                     │
│       │                           │ 绑定钱包 │                     │
│       │                           │ 删除Nonce│                     │
│       │ { success: true }         └──────────┘                     │
│       │◄──────────────────────────                                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 安全措施清单

| 风险 | 防护措施 | 实现方式 |
|------|----------|----------|
| **重放攻击** | Nonce 一次性使用 + 过期时间 | 签名验证后立即删除 Nonce，5分钟过期 |
| **签名伪造** | 服务端验证签名恢复地址 | ethers.verifyMessage 验证 |
| **地址冒用** | 绑定时必须签名证明所有权 | personal_sign 签名验证 |
| **并发绑定** | 数据库唯一约束 + 事务锁 | user_wallets.wallet_address UNIQUE |
| **中间人攻击** | HTTPS 强制加密 | 生产环境强制 HTTPS |
| **金额篡改** | 链上交易验证 | 服务端监控链上事件，不信任前端数据 |

### 4.3 Nonce 生成规则

```javascript
// Nonce 格式
{
  nonce: "0x" + randomBytes(32).toString('hex'),
  createdAt: timestamp,
  expiresAt: timestamp + 300000, // 5分钟
  purpose: "wallet_bind",        // 用途标识
  userId: "xxx"                  // 请求用户
}
```

## 五、数据库设计

### 5.1 user_wallets 表（已存在）

```sql
CREATE TABLE user_wallets (
  id SERIAL PRIMARY KEY,
  user_id INTEGER NOT NULL REFERENCES users(id),
  wallet_address VARCHAR(42) NOT NULL,  -- 小写存储
  wallet_type VARCHAR(20) NOT NULL,      -- 'metamask', 'coinbase', 'walletconnect'
  is_primary BOOLEAN DEFAULT TRUE,
  is_verified BOOLEAN DEFAULT FALSE,     -- 新增：是否已验证
  bound_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  last_used_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  UNIQUE (wallet_address),               -- 一个地址只能绑定一个用户
);

-- 一个用户只能有一个主钱包
CREATE UNIQUE INDEX idx_user_primary_wallet 
ON user_wallets(user_id) WHERE is_primary = TRUE;
```

### 5.2 wallet_bind_nonces 表（新增）

```sql
CREATE TABLE wallet_bind_nonces (
  id SERIAL PRIMARY KEY,
  user_id INTEGER NOT NULL REFERENCES users(id),
  wallet_address VARCHAR(42) NOT NULL,
  nonce VARCHAR(66) NOT NULL UNIQUE,
  purpose VARCHAR(20) NOT NULL,          -- 'bind', 'change'
  expires_at TIMESTAMP NOT NULL,
  used BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_nonce_expires ON wallet_bind_nonces(expires_at);
```

## 六、API 设计

### 6.1 钱包绑定 API

#### 获取绑定 Nonce
```
POST /api/wallet/bind/nonce
Authorization: Bearer <token>

Request:
{
  "address": "0x..."
}

Response:
{
  "nonce": "0x...",
  "message": "请使用钱包签名此消息以完成绑定",
  "expiresIn": 300
}

Error:
{
  "error": "该地址已被其他账户绑定"
}
```

#### 验证并绑定
```
POST /api/wallet/bind/verify
Authorization: Bearer <token>

Request:
{
  "address": "0x...",
  "signature": "0x...",
  "walletType": "metamask"
}

Response:
{
  "success": true,
  "wallet": {
    "address": "0x...",
    "boundAt": "2024-..."
  }
}

Error:
{
  "error": "签名验证失败" | "Nonce已过期" | "该地址已被绑定"
}
```

#### 查询绑定状态
```
GET /api/wallet/bind/status
Authorization: Bearer <token>

Response:
{
  "isBound": true,
  "wallet": {
    "address": "0x...",
    "walletType": "metamask",
    "boundAt": "2024-...",
    "shortAddress": "0xf0fc...29e3"
  }
}
```

#### 更换绑定地址
```
POST /api/wallet/bind/change
Authorization: Bearer <token>

Request:
{
  "newAddress": "0x...",
  "signature": "0x..."
}

Response:
{
  "success": true,
  "wallet": {
    "address": "0x...",
    "boundAt": "2024-..."
  }
}
```

## 八、边界情况处理

### 8.1 钱包绑定场景

| 场景 | 处理方式 |
|------|----------|
| 用户取消签名 | 提示"已取消"，允许重试 |
| 签名超时 | Nonce 5分钟过期，需重新获取 |
| 地址已被其他用户绑定 | 提示"该地址已被绑定"，不允许绑定 |
| 用户已有绑定地址 | 提示"您已绑定地址，如需更换请点击更换" |
| 网络错误 | 提示网络错误，允许重试 |
| 钱包未安装 | 引导安装钱包扩展 |

### 8.2 充值场景

| 场景 | 处理方式 |
|------|----------|
| 充值金额低于最低限额 | 提示"最低充值 1 USDT" |
| 转账到错误地址 | 无法追回，提示用户仔细核对 |
| 交易失败/回滚 | 不增加筹码，显示失败状态 |
| 同一笔交易重复提交 | 幂等处理，只增加一次筹码 |
| 用户更换绑定地址 | 历史充值记录保留，新充值使用新地址 |
| 充值未到账 | 用户可提交 TX Hash 手动查询 |

### 8.3 并发场景

| 场景 | 处理方式 |
|------|----------|
| 两个用户同时绑定同一地址 | 数据库唯一约束，先到先得 |
| 用户多设备同时操作 | Nonce 绑定用户ID，防止跨用户使用 |
| 签名验证并发请求 | 使用数据库事务锁 |
